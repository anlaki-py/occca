# Fix: Shell escape (! command) should save to context memory

## Problem
When users execute shell commands using the `!` prefix (e.g., `!ls -la`), the command and its output are NOT saved to the agent's message history. This means the model has no awareness of:
1. What commands the user has executed
2. What output those commands produced

This is inconsistent with how normal agent tool calls work - when the agent uses the Bash tool, the command and output are stored in `this.messages`.

## Current Behavior
In `src/entrypoints/cli.ts` lines 206-212:
```typescript
// Handle shell escape (! prefix)
if (trimmed.startsWith('!')) {
  const { executeBash } = await import('../tools/BashTool/index.js');
  const result = await executeBash({ command: trimmed.slice(1).trim() });
  console.log(result);
  continue;
}
```

The command runs, output prints to console, but nothing is added to `agent.messages`.

## Solution
Add a method to the Agent class to inject shell command execution context, then call it from the shell escape handler.

### Changes Required

#### 1. Add method to Agent class (`src/agent.ts`)
Add a new public method `addShellCommand(command: string, output: string)` that adds user and assistant messages recording the shell command execution.

#### 2. Update shell escape handler (`src/entrypoints/cli.ts`)
After executing the bash command, call the new agent method to save the command and output to context.

## Implementation Details

### File: `src/agent.ts`
Add new method after `clearHistory()`:

```typescript
/**
 * Add a shell command execution to the message history.
 * This allows the model to see commands the user ran via ! escape.
 * @param command - The shell command that was executed
 * @param output - The output from the command
 */
addShellCommand(command: string, output: string): void {
  this.messages.push(
    { role: 'user', content: `[Shell command executed]: !${command}` },
    { role: 'assistant', content: `[Output]:\n${output}` }
  );
}
```

### File: `src/entrypoints/cli.ts`
Update lines 206-212:

```typescript
// Handle shell escape (! prefix)
if (trimmed.startsWith('!')) {
  const command = trimmed.slice(1).trim();
  const { executeBash } = await import('../tools/BashTool/index.js');
  const result = await executeBash({ command });
  console.log(result);
  agent.addShellCommand(command, result);
  continue;
}
```

3. Ask the model: "What commands have I run?"

# Fix: Terminal input jank after shell escape command

## Problem
After running a shell escape command (`!ls`), the terminal becomes unresponsive. The user must press Enter before they can type the next input. This creates a janky UX where the prompt appears but keystrokes aren't registered.

## Root Cause
In the REPL loop (`src/entrypoints/cli.ts`), each iteration:
1. Creates a new `readline.Interface`
2. Calls `rl.question()` to get input
3. Closes the readline in the `finally` block

After executing a shell command with `console.log(result)`, the loop `continue`s and creates a new readline interface. The issue is that the terminal TTY state may not be properly synchronized - the previous readline's cleanup and the new readline's initialization may leave stdin/stdout in an inconsistent state.

Specifically, when `rl.close()` is called, it may leave residual input buffering or the terminal may need a flush before accepting new input.

## Solution
Add a small delay or explicit stream drain after outputting the shell command result before continuing the loop. This gives the terminal time to stabilize.

## Implementation

### File: `src/entrypoints/cli.ts`

Update the shell escape handler (lines 206-214):

```typescript
// Handle shell escape (! prefix)
if (trimmed.startsWith('!')) {
  const command = trimmed.slice(1).trim();
  const { executeBash } = await import('../tools/BashTool/index.js');
  const result = await executeBash({ command });
  console.log(result);
  agent.addShellCommand(command, result);
  // Wait for stdout to flush before next readline prompt
  await new Promise(resolve => process.stdout.once('drain', resolve));
  continue;
}
```

The `process.stdout.once('drain')` ensures the output buffer is fully written before continuing. If the buffer is empty, 'drain' fires immediately.

**Alternative (simpler):** If the drain approach doesn't work, use a small timeout:
```typescript
await new Promise(resolve => setTimeout(resolve, 50));
```

2. Run: `!echo test`
3. Immediately try typing the next command - should work without pressing Enter first
4. Run: `!ls` (larger output)
5. Immediately type - should work smoothly

# Fix: Terminal completely broken after shell escape - prompt stuck, requires Enter to unfreeze

## Problem
After running `! npm -v`, the user sees:
```
❯ ! npm -v
11.10.0
❯
```
At this point:
1. Cannot type anything
2. Must press Enter to unfreeze
3. After pressing Enter, the `❯ ` prompt disappears
4. User can then type, but the prompt reappears only after typing and deleting some text

This is a severe terminal state corruption issue.

## Root Cause Analysis

### The REPL Loop Flow
```typescript
while (true) {
  const rl = createReadline();  // Creates NEW readline each iteration
  input = await question(rl, promptStr);  // Shows prompt, waits for input
  rl.close();  // Closes readline in finally block
  // ... process input ...
  // For shell escape:
  console.log(result);  // Outputs command result
  continue;  // Loop back to create NEW readline
}
```

### The Problem: readline.close() + console.log() Race Condition

When you run `! npm -v`:

1. User types `! npm -v` and presses Enter
2. `rl.close()` is called in the `finally` block (line 188)
3. The shell command executes via `executeBash()`
4. `console.log(result)` outputs `11.10.0\n`
5. Loop continues, creates a NEW readline interface
6. `rl.question(promptStr)` writes the prompt `❯ `

**The issue**: `rl.close()` (line 188) happens BEFORE the shell command runs. This closes stdin/stdout handlers. Then:
- The shell command's child process uses `exec()` which creates its own stdio
- After `exec()` completes, `console.log(result)` writes to stdout
- But stdout may be in a weird state because the previous readline was closed

**Even worse**: The `process.stdin.setRawMode?.(false)` at line 175 sets stdin to "cooked" mode (line buffering). This interacts poorly with readline's expectations.

### Why pressing Enter "fixes" it

When the terminal is stuck:
1. stdin is in cooked mode, waiting for a newline
2. The prompt `❯ ` was written, but stdin buffer is empty
3. Pressing Enter sends `\n` to stdin, which unblocks the `rl.question()` call
4. readline receives empty string, returns, loop continues
5. But now the prompt state is desynced - readline thinks it already showed the prompt, but the terminal display is corrupted

### The Real Issue: Creating/Closing readline Every Iteration

Node.js readline is designed to be persistent. Creating and closing it repeatedly is an anti-pattern that causes:
1. Terminal state thrashing
2. History not persisting properly between iterations
3. Signal handler conflicts
4. Buffer/state corruption

## Solution: Single Persistent Readline

Instead of creating/closing readline every iteration, create ONE readline instance at the start and reuse it throughout the REPL session.

### Implementation

#### File: `src/entrypoints/cli.ts`

Replace the loop structure:

**Before:**
```typescript
while (true) {
  const rl = createReadline();
  let input: string;
  try {
    input = await question(rl, promptStr);
  } catch {
    console.log('');
    printInfo('Goodbye!');
    process.exit(0);
  } finally {
    rl.close();
  }
  // ... process input ...
}
```

**After:**
```typescript
const rl = createReadline();

try {
  while (true) {
    let input: string;
    try {
      input = await question(rl, promptStr);
    } catch {
      console.log('');
      printInfo('Goodbye!');
      process.exit(0);
    }

    const trimmed = input.trim();
    if (!trimmed) continue;

    saveHistoryLine(trimmed);

    // Handle slash commands
    if (trimmed.startsWith('/')) {
      const shouldExit = await handleCommand(trimmed, agent, config);
      if (shouldExit) {
        rl.close();
        process.exit(0);
      }
      continue;
    }

    // Handle shell escape (! prefix)
    if (trimmed.startsWith('!')) {
      const command = trimmed.slice(1).trim();
      const { executeBash } = await import('../tools/BashTool/index.js');
      const result = await executeBash({ command });
      console.log(result);
      agent.addShellCommand(command, result);
      continue;
    }

    // Run agent turn
    try {
      await runAgentTurn(agent, trimmed);
    } catch (err: any) {
      printError(formatApiError(err));
    }
  }
} finally {
  rl.close();
}
```

### Key Changes

1. **Move readline creation OUTSIDE the loop** - Create once, reuse forever
2. **Remove `finally { rl.close() }` from inside the loop** - Only close on exit
3. **Close readline on explicit exit** - Call `rl.close()` before `process.exit(0)` in exit paths
4. **Remove the drain hack** - No longer needed since we're not thrashing readline state

### Why This Fixes The Issue

1. **Stable terminal state**: readline manages the terminal consistently without open/close thrashing
2. **Proper prompt handling**: readline's internal cursor and prompt state stay synchronized
3. **No stdin/stdout corruption**: The streams remain in a consistent state throughout the session
4. **History works properly**: readline's internal history buffer persists naturally
